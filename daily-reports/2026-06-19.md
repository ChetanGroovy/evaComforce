## 0. The big architectural decision (locked)

**BRIDGE architecture — Python is the brain, TypeScript is the face.**

- Our **Python engine** runs as an HTTP service (the "brain") and owns the whole screening
  conversation: greeting, warm phrasing, KB answers, free-text understanding, BMI, rules.
- Chetan's **comforce_v2 TS** is the "face": his React UI + question-generation, plus a
  **thin proxy** that forwards screening calls to the brain.
- Rejected earlier approaches: (a) re-implementing our engine in TypeScript (caused two
  copies / drift), (b) a Python server that also serves the React UI (`ui_server.py`).
- Single source of truth for clinical logic = the Python engine.

```
React UI ──> TS API (Fastify, :7765) ──proxy──> Python brain (:7801) ──> our Engine
                │
                └─ studies list / detail / +New Study  (stays Chetan's TS)
```

## 2. ⚠️ OPEN ISSUE — study-format / BMI regression (NOT fixed)

Our Python `flow.py` only implements a **subset** of Chetan's `study.json` schema. His
pipeline emits rich studies using features our engine does NOT handle:
- `show_if` (conditional questions), `routing`, and a **merged `bmi` answer_type**.

Concrete problems found (evidence in `studies\*\study.json`):
- **AZD0780** uses `show_if: "sex_at_birth == \"Female\""` → our engine ignores `show_if`
  and would ask a **male patient "Are you pregnant?"** and mis-handle that exclusion.
- **WC45276** `study.json` has one merged `q_bmi` (answer_type `bmi`); our `flow.py` computes
  BMI from **two `height_weight` questions**, so reading `study.json` it **SKIPS the BMI gate**
  (low-BMI patients wrongly pass). WC45276 also has the correct legacy `questions.json`.

**State:** Because `config.py` default now points at `evaComforce\studies` (where WC45276 has
`study.json`) and `flow.py` still **prefers `study.json`**, WC45276 BMI is currently broken on
the brain. The proposed fix — flip `flow.py` to prefer `questions.json` when present — was
**reviewed but REJECTED by the user**, so `flow.py` is UNCHANGED. This must be resolved.

The deeper open question (parked): **whose engine decides eligibility for Chetan's rich
studies?** Options discussed: (a) extend Python `flow.py` to fully match his schema, (b) let
Chetan's TS engine screen and Python only does conversation, (c) restrict studies to the

## 3. Humanize feature (built, NOT wired)

Goal: questions Chetan's pipeline generates read bookish/clinical; rewrite them
patient-friendly **without touching machine fields** (variable_name, answer_type,
disqualify_condition, criteria_ids, show_if, rank).

- **Built**: `POST /humanize` on the brain + `agent.humanize_question()` (uses our LLM via
  Claude Code CLI, no key; offline = unchanged). Idempotent (stashes `sms_question_raw`).
- **Tested**: live dry-run on AZD0780 (`backend=claude_code`, 8 questions) — good rewrites,
  nothing written to disk.
- **NOT done / pending decisions:**
  - User wants the humanized list saved as a **NEW `refinedQuestions` array**, leaving
    `screeningQuestions` untouched. Current code mutates `screeningQuestions` in place — **needs
    changing to emit `refinedQuestions`**. (Note: `flow.py` already prefers `refinedQuestions`
    when present, so screening would use humanized wording.)
  - **Not wired** into any pipeline (it's inert until called).

### Pipeline reality (why humanize isn't auto-triggered)
The "+New Study → Upload & Create" button (`apps/web/.../NewStudyModal.tsx` →
`apps/api/.../studies.ts createStudy`) ONLY: saves PDFs, runs `pdftotext` →
`protocol.txt`/`icf.txt`, and writes a **DRAFT `study.json` with `screeningQuestions: []`**.
**There is NO automated criteria/question/KB generation** — that's currently a manual Claude
step (`STUDY-INPUT-CONTRACT.md`: "I (Claude) extract everything") and is the part **Chetan is
still building**. So there's nothing to wire humanize "after" yet.

**Parked decision** (user said do it later): who builds the auto-generation (criteria →
questions → KB) so the whole thing runs from one button — our Python brain (`POST /onboard`)
vs Chetan's TS. Until then, humanize stays standalone.
