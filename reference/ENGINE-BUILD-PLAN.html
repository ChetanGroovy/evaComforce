1Executive summary

The live Alleviate platform prescreens patients with an LLM walking a flow graph: it asks a question, reads the free-text reply, and the LLM both understands the answer and chooses the branch. That is simple to author but non-deterministic and unauditable.

Our engine takes the hybrid path: the LLM only extracts the answer into a structured value; a deterministic code path decides every branch and the final verdict. Same conversational experience for the patient; reproducible, explainable, regulator-grade decisions for us.

2Goals & non-goals

Goals

  • Prescreen a patient against a study's questionnaire deterministically — same answers always yield the same verdict.
  • Match the live agent's conversational feel (warm intro, name, acknowledgments, height/weight→BMI, scheduling close).
  • Emit a full audit trace and protocol traceability (each question → criterion → protocol page) for every decision.
  • Three honest terminals: QUALIFIED (pre-qualified), DNQ (specific reason), INCOMPLETE (not fully screened).
  • Run the same decision core across channels: CLI now, SMS/voice later, behind one adapter interface.
  • Be safe: never qualify a patient who should be excluded.

Non-goals

  • Deciding patient intent ("not interested") — that is conversation management, not eligibility.
  • Replacing the on-site screening visit — labs, imaging, exams, med reconciliation stay at the visit.
  • Authoring criteria automatically — extraction from Protocol/ICF is a separate, existing pipeline.

4Architecture

One decision core, many channels. The LLM is confined to the extraction step; the verdict is pure code.

┌────────────── one conversation turn ──────────────┐ study.json ──compile──► flow │ ask node.prompt → patient free-text reply │ (screeningQuestions) │ │ │ │ LLM / rule EXTRACTOR (fuzzy) │ │ "I'm 52" ─► { q1_age: 52 } │ │ │ │ │ DETERMINISTIC SELECTOR (pure code, no LLM) │ │ evaluate disqualify_condition / show_if │ │ │ │ │ next question · or · terminate │ └─────────────────────────────────────────────────────┘ terminal: QUALIFIED (pre-qualified) | DNQ — <reason> | INCOMPLETE always emits: decision trace + transcript + deferred[] + criteria_ids

Components

ComponentResponsibilityDeterministic?
Compilerstudy.json → ordered, conditioned question modelyes
Extractor (adapter)free text → structured value; confidence / re-ask signalno (LLM/rule) — isolated
Selector / verdictapply show_if, disqualify_condition, routing, defer → terminalyes

Why hybrid. Confining the LLM to extraction means an LLM mistake is localized to one variable and shows up in the trace; the eligibility decision stays reproducible and reviewable. Alleviate's all-LLM design cannot offer this.

6Data model & schemas

Screening question (the executable unit)

{
  "rank": 11,                               // ask order
  "variable_name": "q10_pregnancy",         // key the extractor fills
  "sms_question": "Are you currently pregnant, breastfeeding, …?",
  "answer_type": "yes_no",                  // yes_no | number | choice
  "choices": ["Female","Male"],             // for choice
  "routing": true,                          // collects a value, never disqualifies (e.g. sex)
  "show_if": "sex_at_birth == \"Female\"",  // conditional gate
  "disqualify_condition": "answer == yes",  // the knockout rule
  "capture": "bmi", "bmi_cutoff": 27,       // ask height+weight, compute BMI (conversational)
  "defer_if_unanswered": true,              // unasked → pre-qualify, confirm at visit
  "criteria_ids": ["EXC-37"]                // traceability to protocol criterion
}

Criterion classification (the accuracy keystone)

Every criterion carries verification_method ∈ {self_report, exam, lab, imaging, records, derived}, knockout_strength ∈ {hard, soft, none}, and phone_screenable = self_report && hard. Only phone-screenable criteria may source a phone question; the eval gate fails the build otherwise. Example: GLP-1 (EXC-30) was reclassified to records / not-phone-screenable to match the live agent, which confirms it at the visit.

Conversation record (persisted output)

{
  "patient": "Albert_Warren", "study": "WC45726",
  "terminal": "QUALIFIED",                  // QUALIFIED | DNQ | INCOMPLETE
  "reason": "Pre-qualified; confirm at visit: q10_pregnancy",
  "deferred": ["q10_pregnancy"],
  "criteria_ids": [],                       // for a DNQ, the failed criterion
  "answers": { "q1_age": 52, "sex_at_birth": "Male", … },
  "trace": [ { "rank":1, "variable":"q1_age", "answer":52, "shown":true, "disqualified":false }, … ],
  "transcript": [ { "agent":"…" }, { "patient":"…" }, … ]
}

7Engine internals (verdict logic)

  1. Questions run in rank order.
  2. show_if false → question skipped entirely (not asked, not counted).
  3. routing question → collects a value, never disqualifies (e.g. sex_at_birth gates pregnancy).
  4. Answered + disqualify_condition true → DNQ (first knockout in rank order wins; conversational engine exits early with a polite close).
  5. Unanswered + defer_if_unanswered → deferred (does not block; confirm at visit).
  6. Unanswered + not deferred → missing.
  7. Terminal: any DNQ → DNQ · else any missing → INCOMPLETE · else → QUALIFIED (deferred items noted).
Conversational extras (presentation only — never change the verdict): warm personalized intro, rotating acknowledgments ("Got it."), height+weight→BMI capture, off-topic redirect, scheduling close on QUALIFIED, "stop" to pause. Because these don't touch the decision, conversational ≡ batch is provable (test-converse).

8Extraction layer (LLM)

Today: a rule-based extractor handles yes_no (negation-aware), number, and choice — sufficient for the current question types, with re-ask on ambiguous input.

Production: a per-turn LLM extractor adapter (the screening agent logic) for genuinely free-form replies. Validated in batch on the 20-patient sample: it handled off-topic chatter, computed BMI from height/weight, and caught a "type 1 not type 2" self-correction.

  • Input: question (variable, type, choices) + the patient's raw reply (+ short context).
  • Output: { value, confidence, needs_clarification }.
  • Low confidence → engine re-asks with a clarifying prompt; the branch logic never sees ambiguity.
  • Golden fixtures: free-text → expected value, including adversarial/ambiguous replies.
Determinism is preserved: the extractor feeds values; the verdict is still pure code. An extractor error is visible in the trace and localized to one variable.

The contradiction metric (key safety measure)

Because the platform books after the core questions and defers pregnancy/GLP-1 to the call, raw "agreement" is misleading. We measure contradictions instead: (a) we DNQ a patient the platform booked, or (b) we QUALIFY a patient the platform clinically rejected. Target: 0 type-(b) contradictions (never pass someone who should fail). Type-(a) cases are reviewed — often the engine is correctly stricter (e.g. a disclosed pregnancy).

11Security, privacy & compliance

Patient data is PHI. Nothing below is optional before live patient contact.

AreaRequirement
PHI handling (HIPAA)Encrypt at rest + in transit; access controls + audit logging; BAAs with any LLM/SMS/cloud vendor; minimum-necessary data to the extractor.
LLM vendorUse a HIPAA-eligible API with a signed BAA; no PHI to non-covered endpoints; redact identifiers where possible.
SMS / outreach (TCPA)Prior express consent before texting; honor STOP/opt-out immediately; quiet-hours; per-message opt-out language.
ConsentThis is a pre-screen, not informed consent for the study; ICF consent happens on-site. State that clearly.
AuditEvery conversation persisted with timestamps, full trace, and verdict — tamper-evident, retained per policy.
Secrets.env gitignored; rotate credentials after use; secrets manager in production (never in repo).

Hard gate: a security + compliance review (PHI flow, consent, opt-out, audit, BAAs) must pass before any real patient is contacted on SMS/voice. This is a go/no-go, not a checkbox.

12Deployment & infrastructure

  • Engine core — packaged as a service/module; pure functions, no external state in the decision path.
  • SMS path — Twilio webhook → API endpoint → engine turn → reply; per-patient session store (Redis/DB) keyed by phone + study.
  • Study config — study.json as source of truth; versioned + published (mirrors Alleviate's "Published v11").
  • Extractor — LLM API behind a BAA, with timeout + fallback to re-ask.
  • Persistence — conversation records + verdicts in an encrypted store; export to CRM funnel.
  • Environments — dev / staging (synthetic data) / prod (PHI); no PHI outside prod.
  • CI — run eval, golden, test-converse, and extractor goldens on every change; block deploy on failure.

13Observability & monitoring

  • Funnel by terminal — QUALIFIED / DNQ / INCOMPLETE counts + DNQ-reason breakdown per study (Alleviate parity).
  • Extractor health — re-ask rate, low-confidence rate, clarification loops per question (flags ambiguous wording, e.g. the double-barreled weight-loss item).
  • Drift — periodic transcript replay vs live outcomes; alert if contradiction rate rises.
  • Latency / errors — per-turn LLM latency, webhook failures, session timeouts.
  • Deferred items — track how many pre-qualifications carry deferred questions (pregnancy) for visit follow-up.
